Αυτοματοποιημένο DevSecOps Pipeline με Git Hooks και Docker
Στόχος της εργασίας

Στόχος της εργασίας είναι η σχεδίαση και υλοποίηση ενός αυτοματοποιημένου DevSecOps pipeline, το οποίο θα ενεργοποιείται κατά τη διαδικασία ανάπτυξης και διαχείρισης κώδικα μέσω Git. Το pipeline θα εκτελεί ελέγχους ασφάλειας και ποιότητας κώδικα τόσο σε στατικό επίπεδο, δηλαδή πάνω στον πηγαίο κώδικα, όσο και σε δυναμικό επίπεδο, δηλαδή πάνω στην εκτελέσιμη εφαρμογή ή στο παραγόμενο containerized περιβάλλον.

Η εργασία αποσκοπεί στην εξοικείωση με βασικές πρακτικές DevSecOps, όπως:

    αυτοματοποίηση ελέγχων ασφάλειας,
    ενσωμάτωση security checks σε Git workflows,
    χρήση CI/CD εργαλείων,
    εκτέλεση εργαλείων ασφάλειας μέσα από containers,
    τεκμηρίωση και αξιολόγηση ευρημάτων ασφάλειας.

Περιγραφή εργασίας

Καλείστε να δημιουργήσετε ένα αυτοματοποιημένο DevSecOps pipeline το οποίο θα εκτελείται όταν πραγματοποιείται commit ή push σε Git repository. Για την ενεργοποίηση του pipeline θα πρέπει να χρησιμοποιηθούν Git hooks, ενώ για την ενορχήστρωση των βημάτων του pipeline θα πρέπει να χρησιμοποιηθεί ένα εργαλείο CI/CD, όπως το Jenkins.

Το pipeline θα πρέπει να περιλαμβάνει:

    στατικούς ελέγχους στον πηγαίο κώδικα,
    δυναμικούς ελέγχους στην εφαρμογή ή στο παραγόμενο εκτελέσιμο/container,
    παραγωγή αναφορών με τα αποτελέσματα των ελέγχων,
    τεκμηρίωση της αρχιτεκτονικής και των επιλογών υλοποίησης.

Όλα τα εργαλεία που θα χρησιμοποιηθούν θα πρέπει, όπου είναι εφικτό, να εκτελούνται μέσα από Docker containers.

Οι εργασίες μπορούν να παραδοθούν ατομικά ή σε ομάδες μέχρι τριών ατόμων.

Επιλογή εφαρμογής

Η εφαρμογή που θα χρησιμοποιηθεί μπορεί να είναι:

    εφαρμογή από άλλη εργασία του μεταπτυχιακού,
    open-source project,
    δική σας εφαρμογή,
    εφαρμογή για την οποία έχετε δικαίωμα χρήσης και τροποποίησης.
    Η εφαρμογή μπορεί να είναι γραμμένη σε οποιαδήποτε γλώσσα προγραμματισμού, όπως Python, Java, JavaScript, PHP, C#, Go ή άλλη, αρκεί να μπορούν να εφαρμοστούν σε αυτήν κατάλληλοι έλεγχοι ποιότητας και ασφάλειας.

Στην αρχική έκδοση της εφαρμογής θα πρέπει να εισαγάγετε σκόπιμα ορισμένα προβλήματα ασφάλειας, ώστε να είναι δυνατή η αξιολόγηση του pipeline. Ενδεικτικά παραδείγματα:

    hardcoded passwords ή API tokens,
    ευπάθεια SQL injection,
    ευπάθεια command injection,
    χρήση παρωχημένων ή ευάλωτων dependencies,
    μη ασφαλής διαχείριση αρχείων ή εισόδου χρήστη,
    εκτεθειμένα credentials σε configuration files,
    ευάλωτο Docker image ή λανθασμένο Dockerfile configuration.

Τα προβλήματα αυτά θα πρέπει να είναι ελεγχόμενα και να χρησιμοποιηθούν αποκλειστικά για εκπαιδευτικούς σκοπούς.

Προσοχή! Οι ευπάθειες που θα εισαχθούν στην εφαρμογή θα πρέπει να χρησιμοποιηθούν μόνο σε ελεγχόμενο και απομονωμένο περιβάλλον. Δεν επιτρέπεται η εκτέλεση επιθετικών εργαλείων σε συστήματα τρίτων ή σε εφαρμογές για τις οποίες δεν υπάρχει ρητή άδεια ελέγχου.

Όλοι οι δυναμικοί έλεγχοι θα πρέπει να εκτελούνται αποκλειστικά στην εφαρμογή της εργασίας, σε τοπικό περιβάλλον ή σε περιβάλλον που ελέγχετε πλήρως.
Απαιτήσεις υλοποίησης
1. Git workflow και Git hooks

Θα πρέπει να υλοποιηθεί workflow με Git hooks. Ενδεικτικά μπορείτε να χρησιμοποιήσετε:

    pre-commit hook για γρήγορους τοπικούς ελέγχους, όπως linting και έλεγχο για secrets,
    pre-push hook για πιο εκτεταμένους ελέγχους πριν την αποστολή αλλαγών,
    post-receive ή webhook-based μηχανισμό για ενεργοποίηση του Jenkins pipeline.

Η επιλογή των hooks θα πρέπει να τεκμηριωθεί. Θα πρέπει να εξηγείται ποιοι έλεγχοι εκτελούνται τοπικά και ποιοι εκτελούνται στο CI/CD περιβάλλον.
2. CI/CD pipeline

Θα πρέπει να υλοποιηθεί pipeline σε Jenkins ή αντίστοιχο εργαλείο CI/CD. Το pipeline θα πρέπει να περιλαμβάνει διακριτά στάδια, όπως:

    Checkout του κώδικα από το Git repository.
    Εκτέλεση στατικών ελέγχων.
    Build της εφαρμογής ή του Docker image.
    Εκτέλεση δυναμικών ελέγχων.
    Συλλογή και αποθήκευση των αποτελεσμάτων.
    Παραγωγή τελικής αναφοράς.

Το pipeline θα πρέπει να αποτυπώνεται σε αρχείο, για παράδειγμα Jenkinsfile, ώστε να είναι επαναλήψιμο και τεκμηριωμένο.
3. Στατικοί έλεγχοι

Το pipeline θα πρέπει να περιλαμβάνει τουλάχιστον τρεις από τις παρακάτω κατηγορίες στατικών ελέγχων:

    έλεγχος για hardcoded credentials, secrets, tokens ή private keys,
    linting ανάλογα με τη γλώσσα προγραμματισμού της εφαρμογής,
    static application security testing,
    dependency vulnerability scanning,
    έλεγχος Dockerfile ή Infrastructure-as-Code αρχείων,
    έλεγχος ποιότητας κώδικα και security smells.

Ενδεικτικά εργαλεία:

    TruffleHog,
    Semgrep,
    Bandit για Python,
    CodeQL,
    SonarQube,
    Snyk,
    γλωσσικά linters, όπως ESLint, Pylint, Checkstyle, PHPStan ή αντίστοιχα.

Δεν απαιτείται να χρησιμοποιηθούν όλα τα εργαλεία. Θα πρέπει όμως να αιτιολογηθεί η επιλογή των εργαλείων που θα χρησιμοποιηθούν.
4. Δυναμικοί έλεγχοι

Το pipeline θα πρέπει να περιλαμβάνει τουλάχιστον δύο από τις παρακάτω κατηγορίες δυναμικών ελέγχων:

* port scanning της εκτελούμενης εφαρμογής ή του container,
* vulnerability scanning σε services ή exposed ports,
* dynamic application security testing,
* έλεγχος για SQL injection σε προκαθορισμένα endpoints,
* έλεγχος Docker images για γνωστές ευπάθειες,
* basic security testing σε web endpoints.

Ενδεικτικά εργαλεία:

    Nmap,
    OWASP ZAP,
     sqlmap,
     Trivy,
     Grype,
     Docker Scout,
     Snyk Container.

Για τον έλεγχο SQL injection, τα endpoints που θα ελεγχθούν θα πρέπει να δηλωθούν σε ξεχωριστό αρχείο, για παράδειγμα endpoints.txt ή targets.yaml.

 
5. Docker και αναπαραγωγιμότητα

Η εφαρμογή και τα εργαλεία ελέγχου θα πρέπει να εκτελούνται, όπου είναι εφικτό, σε Docker containers.

Το repository θα πρέπει να περιλαμβάνει:

    Dockerfile για την εφαρμογή,
    docker-compose.yml, εφόσον χρειάζονται πολλαπλά services,
     οδηγίες εκτέλεσης του pipeline,
    scripts ή configuration files για τα εργαλεία ελέγχου,
    τα απαραίτητα αρχεία για την αναπαραγωγή των αποτελεσμάτων.

Η εργασία θα πρέπει να μπορεί να εκτελεστεί από τρίτο άτομο με σαφείς οδηγίες.
Παραδοτέα

Κάθε ομάδα ή φοιτητής θα πρέπει να παραδώσει:

    Git repository με τον κώδικα της εφαρμογής.
    Git hooks που χρησιμοποιήθηκαν.
    Jenkinsfile ή αντίστοιχο αρχείο CI/CD pipeline.
    Dockerfile και, εφόσον απαιτείται, docker-compose.yml.
    Configuration files των εργαλείων ασφάλειας.
    Αρχείο με τα endpoints που ελέγχθηκαν δυναμικά.
    Αναφορές αποτελεσμάτων από τα εργαλεία ελέγχου.
    Τελική τεχνική αναφορά.

Τελική τεχνική αναφορά

Η τελική αναφορά θα πρέπει να περιλαμβάνει:

    σύντομη περιγραφή της εφαρμογής,
    περιγραφή των σκόπιμα εισαγμένων ευπαθειών,
    περιγραφή του Git workflow,
    περιγραφή της αρχιτεκτονικής του pipeline,
    εργαλεία που χρησιμοποιήθηκαν και αιτιολόγηση επιλογής τους,
    screenshots ή logs από επιτυχημένες εκτελέσεις,
    πίνακα ευρημάτων ανά εργαλείο,
    σύντομη αξιολόγηση των αποτελεσμάτων,
    προτεινόμενες διορθώσεις για τα ευρήματα,
    περιορισμούς της υλοποίησης.
